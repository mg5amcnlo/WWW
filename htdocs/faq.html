What is Madgraph?

MadGraph is a matrix element generator [T. Stelzer and W.F. Long, Phys. Commun. 81 (1994) 357-371]. Given a process, it produces a fortran subroutine that computes the squared amplitudes making suitable calls to the HELAS library. With respect to its first version, now MadGraph

  1. is no longer restricted to the standard model but can be easily extended to other models by editing the files particles.dat and interactions.dat.
  2. produces information on the color structure of QCD amplitudes.
  3. produces information on the structure of the Feynman diagrams that can be used to parameterize the phase space on a diagram-by-diagram basis. This information is used by MadEvent.
  4. is able to find all the subprocess contributing to a process that has initial state protons (or antiprotons) and/or jets in the final state. The user can input either a parton level (sub)process (e.g. u~ d > w+ g) or a "physical" process (p p> w+ j).

What is MadEvent?

MadEvent is a multi-purpose event generator. It uses as input the process-dependent information (matrix elements, subprocesses, and phase space mappings) produced by MadGraph on the user's request. MadEvent exploits a new multi-channel integration  method, named Single-Diagram-Enhanced, that makes the integration on different channels completely independent.

What is HELAS?

HELAS stands for HELicity amplitude subroutines for Feynman diagram evaluations [H. Murayama, I. Watanabe, and K. Hagiwara, 1991]. It is a library of Fortran routines which  enables the user to compute the helicity amplitudes of an arbitrary tree-level Feynman diagram with a simple sequence of CALL SUBROUTINE statements. Here you find the HELAS manual.

How many particles in the final state can a process have?

The web version of MadGraph is configured for up to 8 final state particles . This can be increased (upon request), however, the time required to generate a sub-process will increase with the number of external particles requested.

How long does it take to generate a MadEvent code?

It depends on:

  1. the number of particles;
  2. the number of subprocesses, which are determined by how many "summed over" particles, such as protons and jets, are requested.
For instance, for a simple process, like pp>tt~bb~, just a few seconds. For a more complicate one, like, W+4jets, to which many subprocesses contribute, several minutes. (time on a 700 Mhz, single CPU, Linux machine).

How can I prevent a particle from appearing as an internal line?

Add a / and the name of particle to be excluded at the end of the requested process.
(See also EXAMPLES in the main page).

For example, consider the production of a tt~ pair and bb~ pair from gluon scattering at order alpha_S^2 alpha_EM^2. In this case the input would be
gg > tt~ bb~
QCD=2 QED=2
which includes diagrams where the Z,photon,Higgs appear as internal lines. If the interest is on the QED backgrounds to Higgs production in association with a top-quark pair, with H->bb~, then  the diagrams with the Higgs should be excluded (these are the signal!). To this aim is sufficient to input
gg > tt~ bb~/h
QCD=2 QED=2

How can I select diagrams which have a given particle as an internal s-channel line?

Specify it as an intermediate product.
(See also EXAMPLES in the main page).

For example, consider Higgs production in association with a top-quark pair, and with H->bb~, from gluon scattering. With the following input
gg > tt~ bb~
QCD=2 QED=2
we would include diagrams where also the Z,photon appear as a internal lines. To get rid of these contribution input
gg > h > tt~ bb~
QCD=2 QED=2
which tells MadGraph to consider only diagrams where the Higgs is in a s-channel line.

How do I unpack madevent.tar.gz and run the code on my PC Linux?

1. download madevent.tar.gz into an empty directory.
2. gunzip madevent.tar.gz        : unzip the archive.
3. tar -xvf madevent.tar         : unpack the archive.

The following directory structure will appear:
 
Dir name Content
Events the generated events (weighted and unweighted)
Process info on the process
Source source (fortran) files 
SubProcesses directories containing the various subprocesses
bin executables and scripts (csh and Perl)
lib libraries

together with the HTML file index.html that contains generic information on the code.

4. Run the survey script that makes all the files and starts a quick integration:

./bin/survey 0        (for more info see What do the various scripts in the directory /bin do? )

5. netscape SubProcesses/results.html  : to look at your results.

What do the various scripts in the directory /bin do?

The basic scripts are three: survey, refine, and unweight_events. These are meant to be run one after the other.

survey performs a quick integration over the phase space without adjusting the grids. It asks the user to provide the run mode in parallel (1) or on a single machine (0). When running in parallel (PBS system is supported)  the name of the job has to  be provided.
For instance:
./bin/survey 0       : means that the user has requested to run on a single machine
./bin/survey 1 goofy   : means that the user has requested to run in parallel, giving the jobs name "goofy". The results of this first run (as well as all the others) can be looked up with a browser in the file SubProcesses/results.html.

refine performs an integration based on the information gathered by survey. It adjusts the grids to improve the efficiency. It asks the user to provide the number of desired unweighted events  (~10000) or the percent accuracy of the integration (e.g. 0.01 = 1% accuracy). Then, it asks the run mode, parallel (1) or on one machine (0). When running in parallel (PBS system is supported) the name of the job can be provided.
For instance:
./bin/refine 10000 0    : means that the user has requested to produce enough weighted events that 10000 unweighted events can be extracted,, running on one machine.
./bin/refine 0.01 1 pippo   : means that the user has requested to obtain a cross section with 1% accuracy, running in parallel, giving the jobs name "pippo". The results of this run can be looked up with a browser in the file SubProcesses/results.html. The unweighted events are in events.dat in the Les Houches format.

unweight_events  performs the unweighting of the events. The events are stored in the Les Houches format in the file unweighted_events.dat .

How do I get a quick (and rough) estimate of the cross section?

Before going all the way, and spend (CPU) time in long runs, it is always useful to have an idea of how large is the cross section by performing a quick run. This is also the right time to check that the process, the QCD and QED couplings, the cuts, the collider energy, and all the other settings are what are supposed to be.

To this aim, use the script survey located in the directory /bin. It performs a quick integration over the phase space without adjusting the grids. It asks the user to provide the run mode in parallel (1) or on one machine (0). When running in parallel (PBS system is supported)  the name of the job can be provided.
For instance:
./bin/survey 0       : means that the user has requested  on a single machine
./bin/survey 1 goofy   : means that the user has requested  to run in parallel, giving the jobs name "goofy". The results of this first run (as well as all the others) can be looked up with a browser in the file SubProcesses/results.html.

How do I calculate a cross section at a given precision?
 

Use the script refine, after having used survey, located in the directory /bin. refine performs an integration over the phase space adjusting the grids. It asks the user to provide the run mode in parallel (1) or on one machine (0). When running in parallel (PBS system is supported)  the name of the job can be provided.
For instance:
./bin/refine 0.01 1 pippo  : means that the user has requested to obtain a cross section  with 1% accuracy, running in parallel, giving the jobs name "pippo".

How do I generate unweighted events after I calculated the cross section at a given precision, without wasting the information gathered so far?

First, you can just try running unweight_events, which performs the unweighting of the events, and see how many unweighted events are obtained:
./bin/unweight_events

If they are not enough, run refine asking for 10000 events.
For instance:
./bin/refine 10000 0    : means that the user has requested to produce enough weighted events that 10000 unweighted events can be extracted,, running on one machine.
Finally, run unweight_events to unweight the events.

How do I save/retrieve/remove the results of a run?

Use the scripts save, retrieve, remove the directory /bin, followed by the name that identifies the run.
For instance:
./bin/save tevatron : copies all the results.html pages and the events (and the grids) into tevatron_results and tevatron_events files, etc. It also updates the index.html file in the main directory, so that the relevant information (web pages and events) are accesible from it.
./bin/retrieve  tevatron: copies all the information (results.html, events, grids) to the default (active) run.
./bin/remove  tevatron: deletes all the information (results.html, events, grids) relative to the "tevatron" run.

How do I change the collider type and/or the energy?

The default is a proton-antiproton collider @ 2 TeV (Tevatron).

Edit the corresponding lines in file Source/setpara.f. See the Manual for futher info.

How do I change the cuts?

The default cuts are:

Et(i)      >10 GeV     : min transverse energy
DeltaR(i,j)>0.4        : min DeltaR=Sqrt(DeltaEta^2+DeltaPhi^2)
|Eta(i)|   <2.5        : max rapidity

for all i,j particles in the final state.

Edit the file Source/setcuts.f to change them. The notation is transparent. See the Manual for futher info.

How can I set an event-by-event renormalization or factorization scale?

set_ren_scale and set_fac_scale. See the Manual for futher info.

How do I make plots?

The file Source/plot_events.f, produces a set of basic plot (topdrawer) of an event file. Use the script plot in /bin, followed by the name of the file of events you want to use.

Should I plot the weighted or unweighted events?

In the unweighting procedure information is lost. So in order to get smoother distributions it certainly better to use the weighted set of events. On the other hand, it can also be useful to check that analogous distributions for the unweighted events are compatible one with the other.
